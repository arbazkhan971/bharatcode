# Release gate: Codex-parity

This is the pre-release checklist that protects the Codex-parity work. The goal
of that work is a CLI that *feels* like Codex: a clean transcript, tool activity
that is visible but quiet, concise file-aware and verification-aware final
answers, and an agent that never claims a task is done without verifying it.

Those properties are easy to regress quietly — a refactor of the renderer, a
prompt tweak, or a change to the run summary can erode them without breaking a
single unit test. This document lists the checks that catch such regressions
**before** a tag ships, and the `release-gate` job in
[`ci.yml`](../../.github/workflows/ci.yml) runs the automatable ones on every
push and pull request so a release branch cannot regress them unnoticed.

> **Scope.** This gate guards the parity *behaviour*. The mechanics of cutting a
> release (GoReleaser, the Homebrew tap, the npm publish) live in
> [`release.yml`](../../.github/workflows/release.yml) and are out of scope here.

## What the gate protects

| Parity property | Guarded by | Regresses as |
| --- | --- | --- |
| Clean transcript / no redraw flood | UX smoke (headless TUI byte budget) | Capture balloons with imperceptible frames |
| Visible-but-quiet tool activity | UX smoke + TUI tests | Activity stream goes silent or spams |
| File-aware final answers | UX smoke (`Changed files:` summary) | Edits ship with no change summary |
| Verification-aware finals / "don't claim done without verifying" | Parity eval subset (`verifiedBuildCheck`) | A task passes without an edit **or** without a build/test run |
| `doctor` provider/sign-in honesty | UX smoke (ChatGPT status line) | A broken sign-in stops being diagnosable offline |
| Clean exits, no hangs | UX smoke (version/doctor/json run exit 0) | A command hangs or exits non-zero |

## Automated checks (run by CI)

Three of the four run in the `release-gate` job (#2–#4); the full suite (#1)
runs in the separate `build-test` job. All are offline and deterministic, so
they are safe to gate on and cheap to run on every PR.

> **Note.** The gate's parity-eval and UX-smoke steps depend on files owned by
> sibling work — the eval suite in `internal/eval` (T15) and
> `scripts/ux-smoke.sh` (T16). The gate is inert until those land; confirm they
> are committed before relying on it.

### 1. Full test suite (build-test job)

```sh
go test ./...
```

The baseline, provided by the `build-test` job (not the `release-gate` job).
Includes the parity eval unit tests in `internal/eval`
(`TestRunCodexParitySignalStable`, `TestRunCodexParityIsDeterministic`, …),
which assert the suite's pass/changed-files/verification signal stays stable.

### 2. Parity eval subset (T15)

```sh
go test ./internal/eval/ -run 'CodexParity|FrontendBuild' -count=1 -v
```

Runs only the Codex-parity tasks from `eval.CodexParitySuite()` — the small app
builds (todo/calculator/notes/quiz), the Go/Node bug fixes, and the frontend
build. Each task only passes when the agent **both** edited at least one file
**and** ran a verification command (`verifiedBuildCheck`), so this is the
machine check for "don't claim done without verifying".

The suite is not yet on the `bharatcode eval` CLI (the parity-aware command path
is owned elsewhere — see [`docs/evals/codex-parity.md`](../evals/codex-parity.md));
until it lands, the Go test is the entry point and the one CI gates on. When the
CLI path lands, swap this step for `bharatcode eval --suite codex-parity`.

### 3. UX smoke (T16)

```sh
go build -o bharatcode .
scripts/ux-smoke.sh ./bharatcode
```

Guards the rough edges that surface only in a real CLI/TUI session: the redraw
flood, the `Changed files:` summary, the `doctor` ChatGPT status line, and clean
exits. Provider-dependent checks (live `run`) **skip** rather than fail when no
provider is reachable, so the script is green offline. The gate runs it that
way; set `UX_SMOKE_REQUIRE_PROVIDER=1` with a reachable provider to turn the
skips into hard assertions during a manual pre-release pass.

### 4. Install-method smoke

```sh
# source install — proves `go install` produces a runnable binary
go install .
"$(go env GOPATH)/bin/bharatcode" version

# npm launcher — validates the shim + postinstall downloader parse and load
node --check npm/bin/bharatcode.js
node --check npm/install.js
```

The npm package is a thin launcher whose `postinstall` downloads the release
binary, so the *full* npm path can only be exercised once release assets exist
(post-tag). CI smokes what is checkable offline — that the shim and downloader
are valid and load — and the source install end to end.

### Not automatable in pre-release CI (manual)

- **Homebrew.** The formula is generated by GoReleaser at tag time and pushed to
  the `homebrew-tap` repo, so there is nothing to install until a release exists.
  After a tag publishes, smoke it manually:

  ```sh
  brew install arbazkhan971/tap/bharatcode && bharatcode version
  ```

- **End-to-end npm install.** Once the GitHub release assets are up:

  ```sh
  npm i -g bharatcode-cli@<version> && bharatcode version
  ```

## Manual signoff (pre-tag)

Run before cutting a tag. The automated gate must already be green on the
release branch.

- [ ] `release-gate` job green on the release branch.
- [ ] UX smoke run **with** a reachable provider (`UX_SMOKE_REQUIRE_PROVIDER=1`)
      so the live `run` / `Changed files:` checks assert rather than skip.
- [ ] Eyeball a real TUI session: transcript reads cleanly, tool activity is
      visible but quiet, the final answer names the files it changed and the
      verification it ran.
- [ ] Confirm the agent does not claim success on a task it did not verify.
- [ ] After the tag publishes: Homebrew and npm installs both yield a binary
      whose `version` output matches the tag.
